### 1. Control Plane

This loop handles the communication between the AI and the system's middleware.

**Agent RL $\rightarrow$ Bridge:** `POST /action JSON: {weights: [], scale: +1}`
The PPO Agent sends its calculated action vector to the Bridge. This payload includes the decision to scale (up/down/none) and the specific routing weights for each active container.

**Bridge $\rightarrow$ Agent RL:** `JSON: [{cpu: 0.4, lat: 0.05...}, ...]`
The Bridge responds with the current *Observation Space*. This is an array containing the normalized metrics (CPU, RAM, latency, errors) for every container, allowing the Agent to calculate the reward and plan its next step.


### 2. Infrastructure Management

The Bridge communicates directly with the host OS virtualization layer to manage container lifecycles and retrieve hardware metrics.

**Bridge $\rightarrow$ Docker Daemon:** `Docker API Cmds (Start/Stop/Stats)`
The Bridge uses the Docker SDK (via the UNIX socket) to execute orchestration commands, such as spinning up new instances or gracefully shutting down idle ones.


**Docker Daemon $\rightarrow$ Bridge:** `OS Metrics (cgroups: CPU/RAM)`
The Docker Engine returns hard OS-level metrics derived from *cgroups*, providing the Bridge with accurate CPU usage and memory allocation data for each container.


### 3. L7 Routing & Monitoring

The Bridge interacts with the Proxy to apply the Agent's routing decisions and gather application-layer telemetry.

**Bridge $\rightarrow$ Proxy:** `Update Upstream Weights`
The Bridge dynamically updates the Proxy's configuration (e.g., via Nginx/HAProxy API) to apply the new traffic distribution weights generated by the RL Agent.

**Proxy $\rightarrow$ Bridge:** `Layer 7 Metrics (Latency, HTTP Errors)`
The Bridge queries the Proxy to fetch critical Layer 7 metrics, such as average response times and HTTP 5xx error rates, which are invisible to the Docker Daemon.


### 4. The Data Plane

This is the traditional flow of user traffic through the system.

**Traffic / Users $\rightarrow$ Proxy:** `HTTP Requests (Stress Test)`
Concurrent HTTP requests generated by end-users or stress-testing tools (like Locust) hit the main entry point of the Proxy.

 **Proxy $\rightarrow$ Containers:** `Forwarded HTTP Requests`
The Proxy forwards the incoming requests to the backend Docker containers, distributing the load strictly according to the dynamic weights set by the Control Plane.
